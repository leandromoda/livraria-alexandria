### Sistema de Design — Alexandria

| Atributo | Diretriz |
|---|---|
| **Estética geral** | Editorial/magazine meets luxury bookstore — rico em tipografia, denso em cultura |
| **Paleta dominante** | Tons profundos: azul-noite (#0D1B2A), dourado-âmbar (#C9A84C), marfim (#F5F0E8) |
| **Accent** | Bordô (#8B1A1A) ou sépia (#7B5E3A) para destaques |
| **Tipografia display** | Serifada clássica: Playfair Display, Cormorant Garamond, ou EB Garamond |
| **Tipografia corpo** | Legível e sofisticada: Lora, Crimson Text, ou Source Serif Pro |
| **Evitar** | Inter, Roboto, Arial, gradientes purple-on-white, layouts genéricos de e-commerce |
| **Inspirações** | Powell's Books, Strand Bookstore, Taschen, The New York Review of Books |
| **Motion** | Sutil, elegante — page reveals suaves, hover com profundidade, sem animações agressivas |
| **Layout** | Assimétrico, editorial — quebra de grid intencional, generoso espaço negativo |

## CAPACIDADE DE DESIGN — COMO USAR

Quando solicitado um **componente ou página de UI**, entregar:

1. **Direção conceitual** — estética, tom, diferencial memorável (1 parágrafo)
2. **Layout descritivo** — hierarquia, espaçamento, medidas principais
3. **Estados de interação** — hover, active, disabled, loading
4. **Comportamento responsivo** — breakpoints mobile/tablet/desktop
5. **Acessibilidade** — contraste, foco, aria-labels relevantes
6. **Código completo** — React + Tailwind ou HTML/CSS/JS conforme contexto

### Template de Solicitação de Design

```
Componente/Página: [ex: Hero da Home, Card de Produto, Checkout]
Objetivo do usuário: [o que o usuário quer realizar]
Elementos obrigatórios: [lista de must-haves]
Estilo: [seguir Sistema de Design Alexandria acima, ou variação explícita]
Referências adicionais: [se houver]
```

## EXECUÇÃO DE CÓDIGO — REGRAS

Quando solicitado:

- Entregar código **completo** — 100% preservado + adições incrementais
- Pronto para copiar
- Sem patch parcial, sem trechos isolados

Sempre informar:
- Caminho do arquivo
- Nome do arquivo
- Ponto de alteração

**Nunca:**
- Reduzir número de linhas
- Excluir funções fora do escopo da alteração
- Refatorar além do pedido
- Reorganizar estrutura existente
- Alterar nomenclaturas

---

## REGRA DE ALTERAÇÃO MÍNIMA OBRIGATÓRIA

Ao alterar código:

- Alterar **apenas** o que foi solicitado
- Não modificar trechos não envolvidos
- Não otimizar código existente além do pedido
- Preservar integridade estrutural

---

## ATUALIZAÇÃO DE ESTADO

Quando solicitado **"Atualize o estado"**:

1. Solicitar envio do arquivo de estado atual
2. Aguardar envio completo
3. Registrar apenas alterações desta conversa
4. Manter todo o restante constante
5. Preservar `open_tasks` (exceto concluídas)
6. Aplicar alteração mínima obrigatória

Entrega obrigatória: JSON completo em bloco de código + contagem de linhas (base vs. revisado).
